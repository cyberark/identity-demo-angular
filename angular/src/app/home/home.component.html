CyberArk Identity Demo Web App (Java)

Acme Inc. is an imaginary company using CyberArk Identity APIs, SDKs, and widgets to secure its web applications. This playground application shows all the possible variations that a developer from Acme has at her disposal for this. This app uses Java Spring backend and Angular frontend.

For developer guides and API documentation, please visit CyberArk Identity Developer Program website.
Scenario 1: Acme wants to use its own UI with CyberArk Identity APIs
  • User identities stored in CyberArk Identity
  • Authentication UI built within the Acme demo app
  • Authentication UI uses CyberArk Identity APIs
  • Use cases:
    • Sign-up using Acme UI
    • Authentication (with MFA)
    • User profile updates to CyberArk Identity using APIs
  • Protocol flows demonstrated:
    • OAuth (Authorization code, Authorization code with PKCE, Implicit, Client credentials, Resource owner password grant)
    • OIDC (Authorization code, Authorization code with PKCE, Implicit, Hybrid)
    • Non-standard (just APIs)

Scenario 2: Acme wants to use CyberArk Identity Sign Up/Sign In Widget
  • Using CyberArk Identity Sign Up/Sign In Widget
  • User registers an account with CyberArk Identity server
    • Admin needs to map the role in CyberArk Admin portal for newly registereed user
  • User authenticates to CyberArk Identity server
  • Use cases:
    • Authentication (with MFA) based on the configured policies
    • User profile update using CyberArk Identity APIs
    • Step-up Login Widget for funds transfer
  • Protocols used:
    • OAuth (Authorization code + PKCE) - OIDC access token is not useful in current scenario
    • For other flow, please refer to "Scenario1"
Start
Scenario 3: Acme wants to use CyberArk Identity only for MFA (2FA)
  • User identities stored in Acme demo app
  • Password entry UI built within the Acme demo app
  • MFA (2FA) uses CyberArk Identity MFA widget
  • Use cases:
    • Sign-up using Acme UI
    • Authentication (with MFA)
    • User profile update to Acme database
    • Implicit user profile update to CyberArk Identity using APIs
    • Step-up MFA for funds transfer
  • Protocols used:
    • OAuth (Authorization code + PKCE) - OIDC access token is not useful in current scenario
    • For other flow, please refer to "Scenario1"
